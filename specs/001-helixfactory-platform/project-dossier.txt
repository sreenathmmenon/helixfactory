1. Executive Summary

HelixFactory is an AI-native software factory for enterprise engineering teams.
Its core promise is simple: before an engineer or AI agent changes code, the
platform should understand the codebase, predict what can fail, show blast
radius, enforce human gates for high-risk work, record proof in git, and learn
codebase-specific lessons from every execution.

The product is not intended to be a generic chatbot, code browser, or toy graph.
The target is an enterprise-grade engineering intelligence and governance
platform that sits between code, tickets, pull requests, AI agents, reviewers,
security stakeholders, and audit/compliance teams.

2. Original Aim

The original aim was to build the week-one HelixFactory slice proving the full
platform thesis end to end:

- Ingest a real public repository.
- Build a living code digital twin.
- Query that twin through graph and plain English workflows.
- Run evidence-backed pre-mortems before change begins.
- Visualize blast radius as a contextual graph, never full universe.
- Govern ticket-to-PR style agent execution.
- Require human approval for HIGH and CRITICAL risk.
- Preserve all platform actions as git-native audit evidence.
- Provide architecture Q&A with citations and uncertainty.
- Propose codebase-specific skill refinements after execution outcomes.

The deeper product ambition is to compete with and exceed tools such as
Sourcegraph, GitNexus, graph-based code explorers, AI coding agents, and
compliance workflow tools by combining code intelligence, risk prediction,
execution governance, auditability, and self-learning in one product.

3. Target Users

Primary users:

- Staff engineers and architects who need to understand blast radius before
  approving or implementing changes.
- Engineering managers who want safe AI-assisted delivery, not uncontrolled
  automation.
- Platform teams that need traceability for AI-assisted code changes.
- Security and compliance teams that need proof of what happened, why it
  happened, what risk was known, and who approved it.
- New engineers who need to understand why a system works the way it does.

Enterprise buyer expectations:

- Clear workflow, not a demo screen.
- Cited evidence, not vague AI claims.
- Real codebase support, not toy examples.
- No raw JSON or stack traces in UI.
- Audit records that survive vendor/tool changes.
- Human gates on risky work.
- BYOK AI support so enterprises can use their own model/account boundary.
- Strong visual design, readable typography, and no broken layout states.

4. Product Pillars

4.1 Living Code Digital Twin

Purpose:
Build a queryable model of the codebase with files, functions, classes,
dependencies, ownership, line ranges, entry points, and relationship evidence.

Current implementation:
- Backend ingestion exists.
- Python, TypeScript, and JavaScript scope exists.
- Tree-sitter based parser adapters exist.
- Repository loader filters noisy directories and test/vendor files.
- Twin builder creates nodes and edges.
- Ownership enrichment exists with fallback.
- NetworkX JSON graph persistence exists.
- Repository ingestion API exists.
- Frontend ingestion page exists.

Current status:
Functional week-one implementation exists and backend tests pass. It is not yet
an enterprise-scale twin with incremental updates, semantic data-flow analysis,
cross-repository topology, schema lineage, runtime traces, or deep ownership
from codeowners/org systems.

Production gap:
- Need incremental re-ingestion and graph diff.
- Need stronger relationship taxonomy: data flow, events, runtime dependency,
  infra, schema, API contract, feature flag, and config relationships.
- Need large-repo performance budgets and indexing.
- Need visible ingestion progress per stage and per file class.
- Need codeowners/team ownership integration.

4.2 Change Pre-Mortem

Purpose:
Predict failure before implementation begins. Every finding must be evidence
backed and cite severity, file path, line, dependency chain, owner context, and
preventive check.

Current implementation:
- Proposed change resolver exists.
- Pre-mortem service exists.
- Evidence-backed findings model exists.
- Evidence gaps are surfaced.
- HIGH and CRITICAL findings block auto-approval through approval gates.
- Frontend pre-mortem surface exists.
- Audit commits for pre-mortem exist.

Current status:
Functional for week-one. The system can produce structured findings and gaps,
but depth of prediction is still limited by the current graph evidence and
heuristics around risk.

Production gap:
- Need more precise AST and call-chain evidence.
- Need richer owner context.
- Need preventive checks tied to actual test files, commands, and CI jobs.
- Need confidence and evidence strength model.
- Need pre-mortem comparison across similar historical changes.
- Need a stronger "no evidence, no finding" enforcement UX.

9. Competitive Position

Compared with Sourcegraph:
- Sourcegraph is strong at search, code navigation, and code intelligence.
- HelixFactory aims to go beyond search by predicting risk, enforcing gates,
  recording evidence, and improving agents over time.
- Current gap: Sourcegraph-level code search, scale, indexing, and polish are
  not yet reached.

Compared with GitNexus / graph-oriented tools:
- Graph tools are strong at visual topology and dependency exploration.
- HelixFactory aims to combine graph exploration with change pre-mortem,
  blast-radius risk, execution governance, and audit evidence.
- Current gap: graph layout, search, clustering, path explanation, and large
  graph performance need significant improvement.

Compared with AI coding agents:
- AI agents are strong at implementation speed.
- HelixFactory's differentiator is governance: no code without context,
  evidence, risk classification, approval gates, and audit trail.
- Current gap: real autonomous execution and PR creation are still modeled, not
  fully implemented.

Compared with compliance/audit tools:
- Compliance tools are strong at controls and reporting.
- HelixFactory can be stronger if every engineering action is natively tied to
  code evidence, graph impact, and git audit records.
- Current gap: exportable evidence packages, RBAC, filters, and enterprise
  retention controls are still needed.

10. Next Roadmap to Beat Competitors

Phase A: Make the Twin Enterprise-Real

- Incremental re-ingestion on git diff.
- Graph diff between commits/branches.
- Codeowners/team ownership.
- Cross-repository graph model.
- Strong relationship taxonomy: calls, imports, extends, data_flow, event,
  schema, API, infra, config, feature_flag, runtime_dependency.
- Large repo indexing and performance tests.

Phase B: Make Graph Exploration Best-in-Class

- Instant search with debounce, cancellation, and local result preview.
- Force layout quality tuning with clustering by file/module/service.
- Relationship filters and legend.
- Path explanation panel: "why are these nodes connected?"
- Finding-to-node and node-to-finding links.
- Graph snapshots and graph diff.
- Large graph virtualization/performance budgets.
- Browser screenshot validation across desktop/mobile.

Phase C: Make Pre-Mortem Defensible

- Finding confidence and evidence strength.
- Preventive checks mapped to real tests/commands.
- Similar historical change retrieval.
- Owner and approver resolution.
- Suppress every unsupported claim by default.
- Dedicated "insufficient evidence" UX that tells the user what data is missing.

Phase D: Make BYOK AI Real for Enterprises

- Keep backend .env mode for now per decision.
- Add provider abstraction tests for Anthropic/OpenAI/local later.
- Add prompt/version audit records.
- Add redaction policy before sending evidence to AI.
- Add AI disabled fallback behavior in every workflow.
- Later: settings page with encrypted key storage and org policy controls.

Phase E: Make Governed Execution Real

- Ticket integration: GitHub Issues, Jira, Linear.
- Branch creation and PR creation.
- Test command execution and CI evidence.
- Reviewer lane outputs as evidence-backed comments.
- Approval UI with accountable human reviewer.
- Explicit blocked/unblocked workflow.

Phase F: Make Audit a Board-Level Feature

- Chronological evidence chain per change.
- Exportable evidence package.
- Filters by risk, owner, service, date, approver, regulated data.
- Tamper-evidence explanation.
- Audit failure recovery and partial evidence warnings.
- Compliance dashboard for AI-assisted delivery.

Phase G: Make Self-Learning Measurable

- Reflection from real execution outcomes.
- Skill versioning and rollout controls.
- Before/after measurement: did the skill prevent repeat mistakes?
- Human review queue with ownership.
- Reject generic suggestions with evidence.
- Skill impact dashboard.

Phase H: Make the UI Feel Enterprise-Grade

- Page-by-page design review in browser.
- Stronger typography hierarchy.
- Fewer bulky panels, more operational density.
- No dead controls.
- No invisible disabled states.
- No clipped graph/placeholder elements.
- No giant marketing sections inside workbench pages.
- Every empty state gives next action.
- Every error is human-readable and actionable.
- Mobile layouts validated, not assumed.

11. Current Truth

HelixFactory now has a broad week-one platform foundation:

- Backend services exist for ingestion, graph, pre-mortem, blast radius, Q&A,
  audit, execution boundary, approval gates, and skill refinements.
- Frontend pages exist for the major product surfaces.
- BYOK AI through backend environment configuration exists.
- Tests pass at the backend and frontend component/build level.

However, it is not yet at the level where an enterprise buyer would evaluate it
as a finished product. It is closer to a strong architectural prototype with
several real differentiators. To become competitor-beating, the next work must
focus less on adding more surfaces and more on making the core workflows deep,
fast, evidence-rich, visually polished, and validated on real repositories.

The most important next investment areas are:

1. Twin quality and incremental updates.
2. Graph exploration quality and performance.
3. Defensible pre-mortem evidence.
4. Real ticket-to-PR execution integration.
5. Audit evidence chain depth.
6. Enterprise browser-tested UX polish.
7. Measurable self-learning from real outcomes.
